## Issue #1: Locale-only 変換で Catalog の Tier 構造が contrived

### 観察

dogfooding F.1（high-fidelity, EN→JA Locale 変換 of `superpowers:writing-skills`）と F.2 / F.3 で、Catalog の Tier 1/2/3 構造がやや人工的に感じられた。特に F.3（high-utility）で Tier 3 に何を入れるかが恣意的（純粋 Locale 変換に target-environment-specific 「差別化機能」が概念上存在しない）。

### 根本原因（推定）

Tier 1/2/3 の階層化は **Technical 次元変換**（C# → VB.NET、Python → Go 等、target environment が技術的に異なる）を念頭に設計されている。Locale-only 変換（EN → JA、JA → ZH 等）では:

- Tier 1（ソース対応の equivalents）: 自然に埋まる
- Tier 2（target 環境で de-facto 必須）: 存在しない（Locale には「環境固有の必須機能」概念が薄い）
- Tier 3（差別化機能）: 同上

つまり Locale-only では Tier 1 のみが意味を持ち、Tier 2/3 は forced fill になる。

### 改善案

**Spec 側のアプローチ:** `references/catalog-system.md` に「Locale-only short-circuit」節を追加:

> **Locale-only conversions:** when the only changing dimension is Locale (no Technical, no Agent change), the Catalog is **optional**. If generated, it may contain Tier 1 only (Tier 2/3 marked "N/A — Locale-only conversion"). Phase 0 §X.1 / §X.2 are still executed but the user may select "no catalog" with a one-line rationale.

### 受け入れ基準

- Locale-only の dogfooding を 1 件再実施し、Tier 2/3 forced fill が消えることを確認
- 既存 Technical 変換テスト（vb-winforms-skills 等）への regression が無いことを retro-check で確認

### 優先度

**Medium** — 実用上は Tier 2/3 を空にすれば回避可能だが、spec で明示すれば user の混乱を防げる。

## Issue #2: balanced/high-utility で forced on の Phase 4 が小規模変換で重い

### 観察

dogfooding F.2（balanced）/ F.3（high-utility）で、profile が forced on とする Phase 4 が、EN→JA Locale 変換のような小規模ケースには重く感じられた。Phase 4 は本来:

1. evaluation-scenario の最終化
2. blind subagent dispatch
3. 結果採点
4. 採用 → Phase 2 ループバック

を要する。Locale 変換で blind subagent を立てて検証するコストは、変換のサイズに対して過大。

### 根本原因（推定）

Phase 4 は本来 Technical 変換や Agent 変換の有用性検証を想定して設計された厚い検証層。profile による forced on 制御は v1.0 で導入されたが、変換規模に応じた「軽量モード」がない。

### 改善案

**Spec 側のアプローチ:** `references/verification-stages.md` § Phase 4 に "lite mode" を追加:

> **Phase 4 lite mode (v1.x candidate):** when (a) the conversion is Locale-only AND (b) the source skill has no code blocks AND (c) the user opts for lite mode in Phase 0 §7 footer, Phase 4 may be reduced to:
>
> - 1 scenario (the central case only)
> - 1 blind subagent dispatch
> - Immediate convergence (no iteration loop)
>
> The trade-off: lite mode is faster but less thorough. Adopted additions still pass through the 9-cell approval table.

### 受け入れ基準

- balanced profile + Locale-only + lite mode の dogfooding を 1 件実施
- Phase 4 所要時間が現行 forced on の 1/3 以下になること
- 9 マス表の判定経路は変わらないこと

### 優先度

**Low** — 実用上は user が Locale-only で profile=high-fidelity + opt-out を選択すれば回避可能。ただし「Locale-only かつ utility も気にしたい」ユースケースを救えない。

## Issue #3: 高有用性での C 系追加に累積 drift 上限なし

### 観察

dogfooding F.3 では C × heavy 1 件のみだったが、思考実験として「複数 iteration を経て C × heavy が 5 件、C × moderate が 7 件、累計 12 件取り込まれた」ケースを想定すると、**各単発判定は 9 マス表で正しく user 承認済みだが、累積では明らかに HIGH drift** という状態が成立し得る。

### 根本原因

9 マス承認テーブルは **単発の追加に対する判定** であり、**累積効果を考慮しない**。drift verdict 計算式（`references/drift-report.md`）も総数ベースだが、profile による調整がない。

### 改善案

**Spec 側のアプローチ:** `references/drift-report.md` § Drift verdict criteria に profile-linked 累積閾値を追加:

> **Cumulative threshold by profile (v1.x):**
>
> | Profile | C × heavy 上限 | C 系合計上限 |
> |---|---|---|
> | high-fidelity | 0（C 系は 9 マスで rejected） | 0 |
> | balanced | 1 件超で verdict を一段昇格 | 3 件超で verdict を一段昇格 |
> | high-utility | 3 件超で verdict を一段昇格 | 8 件超で verdict を一段昇格 |
>
> Verdict escalation: LOW → MODERATE → HIGH。閾値超過は drift-report の Rationale に明記。

### 受け入れ基準

- 仮想テスト: 累計 12 件（C×heavy 5 / C×moderate 7）の dogfooding シナリオで verdict が「自動的に HIGH に昇格」することを確認
- 既存 vb-winforms-skills v2.0/2.1（C 系 0 件）は MODERATE 据え置きで影響なし

### 優先度

**Medium-High** — v1.0 を高有用性で運用し始めると顕在化する可能性あり。現行 spec では「単発判定は正しいが累積は問題」という silent failure が起き得る。

## Issue #4: balanced/high-utility opt-out → halt 経路は dogfooding で実走できなかった

### 観察

`references/profile-system.md` §「Opt-out attempted under balanced or high-utility」に halt → 再選択の挙動が記述されているが、dogfooding F.2 / F.3 では forced on を素直に受け入れたため、halt 経路は実走されていない。

### 根本原因

halt 経路は **negative path**（user が間違った選択をしたときの error handling）であり、dogfooding の正常パス（profile 選択 → 規定の振る舞い）では発火しない。E2E 確認には:

1. balanced profile を宣言
2. その上で Phase 4 を opt-out 試行
3. main agent が halt を発動
4. user に再選択を促す
5. user が profile を high-fidelity に変更 or Phase 4 強制 on を受け入れる

の手動操作が必要。dogfooding harness では自動化できない。

### 改善案

**実装側のアプローチ:** smoke test ハーネスを v1.x で構築:

- `tests/smoke-test/profile-opt-out-halt.md` を新規作成
- balanced + opt-out 試行のシナリオを記述
- expected halt メッセージを記録
- user が手動で実走して結果を記録（CI 自動化は v2.x 検討）

### 受け入れ基準

- smoke test ファイル 2 件作成（balanced と high-utility）
- halt メッセージが想定どおり出ることを 1 度実走確認

### 優先度

**Low** — spec 自体は記述済みで、negative path の実装抜けがあっても主流ユースケースは影響しない。安全網としての smoke test。
