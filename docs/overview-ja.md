# skill-conversion とは（わかりやすい解説）

## 一言で言うと

**「既存の Claude Code スキルを、別の言語・ロケール・エージェント向けに、勝手に改変せず変換する」** ためのプラグインです。

## なぜ必要か

Claude Code には素晴らしいスキルが多数公開されていますが、多くは英語・Claude Code 前提・特定言語のコード例で書かれています。これを日本語化したり、別のエージェント (Copilot CLI / Gemini CLI) 向けに移植したり、別のプログラミング言語に対応させたい場面は多々あります。

普通に「このスキルを日本語に翻訳して」と頼むと、以下のような事故が起きます:

- **勝手に情報を追加される**:元には無かった「補足」や「ベストプラクティス」が混入する
- **勝手に情報を削除される**:訳しにくい部分が静かに消える
- **用語が揺れる**:同じ概念が文書内で複数の訳語になる
- **差分が追えない**:どこが変わったのか、なぜ変わったのか後から確認できない

このプラグインは、これらを**構造的に防ぐ**ためのワークフローを提供します。

## 3 つの変換次元

| 次元 | 変えるもの | 具体例 |
|---|---|---|
| **Technical** | プログラミング言語・フレームワーク | C# → VB.NET、Python → Go、React → Vue |
| **Locale** | 自然言語・文字体系 | 英語 → 日本語、英語 → 中国語 |
| **Agent** | 実行エージェント | Claude Code → Copilot CLI、Claude Code → Gemini CLI |

3 つは独立していて、任意の組み合わせが可能です。たとえば「英語 Claude Code 向け TypeScript スキル → 日本語 Copilot CLI 向け Go スキル」という 3 次元同時変換もサポートします。

## 核となる 4 つの仕組み

### 1. Conversion rule tables（変換規則表）

Phase 0 の段階で、次元ごとに「元の構文 → 変換後の構文」の対応表を先に作ります。たとえば Python → Go なら:

| Python | Go |
|---|---|
| `def foo():` | `func foo()` |
| `class Bar:` | `type Bar struct{}` |
| `try/except` | `if err != nil` |

この表をアドホックな判断の前に固めることで、一貫性のある変換ができます。

### 2. Approved-additions register（追加台帳）

元に存在しなかった情報を足すときは、必ずユーザー承認を一件ずつ得て、理由・影響度付きで台帳に記録します:

| 追加日 | 場所 | 要約 | 理由 | 影響度 | 承認者 |
|---|---|---|---|---|---|
| 2026-04-20 | §3.2 | Option Strict On を既定に | VB.NET コード例をコンパイル可能にするため | moderate | oreguchi |

これにより「なぜこの記述が入ったのか」が常に追跡可能になります。

### 3. Fixpoint review loop（不動点レビューループ）

変換直後には、3 つの軸で並列にレビューを行います:

1. **技術チェック**:構文・API・事実が正しいか
2. **文章チェック**:用語が一貫しているか、読みやすいか
3. **元との差分チェック**:register に無い追加や、漏れた情報が無いか

どれか一つでも修正が入ったら、全 3 軸をもう一度回します。**1 回のパスで 3 軸すべて「変更なし」になるまで**繰り返します（重要スキルは 2 回連続）。

### 4. Drift Report（差分レポート）

完了時に、元スキルとの全差分を集計したレポートを生成します:

- Phase 1 承認の追加
- Phase 4 承認の追加
- Phase 4 却下された提案（audit trail）
- 影響度サマリ（minor / moderate / heavy の件数）
- 総合判定 (**LOW** / MODERATE / HIGH)

ユーザーがこのレポートを承認して、ようやく変換完了となります。

## 6 フェーズのワークフロー

```
Phase 0  スコープ宣言          ← 8 項目を先に宣言（次元・規則表・評価シナリオなど）
  ↓
Phase 1  初期翻訳              ← 規則表を機械的に適用
  ↓
Phase 2  Fixpoint ループ       ← 3 軸並列、ゼロ変更まで
  ↓
Phase 3  コード実行可能性検証   ← 変換後コードが実際に動くか
  ↓
Phase 4  経験的チューニング     ← opt-in。ブラインド評価者でスキル発火を測定
  ↓
Phase 5  Auto-fire テスト      ← 新規セッションで description が発火するか
  ↓
完了    Drift Report → ユーザー最終承認
```

## 特徴的な点 — 自分自身で検証済み

このプラグインは、**自分自身を自分の Phase 4 にかけて検証した結果として今の形になっています**。

- 3 シナリオ × 3 iteration = 9 回のブラインド評価実行
  - S1: Python → Go（技術次元のみ）
  - S2: 英語 → 日本語（ロケール次元のみ）
  - S3: 英語 Claude Code → 日本語 Copilot CLI + Go（3 次元同時）
- 全 iteration で精度 100%、retries 0
- Iter3 で収束信号ゼロ達成
- 最終 drift 判定 **LOW**

検証プロセスで判明した SKILL.md の不明瞭点は、個別承認フォームを通じて修正 (F1–F4) または明確化 (U3–U9) され、すべてユーザー承認を経て本体に反映されています。

## 使い方

自然言語で話しかけるだけです:

- 「この C# スキルを VB.NET に変換して」
- 「このスキルを日本語に翻訳して」
- 「この Claude Code スキルを Go で Copilot CLI 向けに移植して」
- 組み合わせ自由（例: 「日本語訳して Gemini CLI 向けに移植」）

プラグインが自動的に Phase 0 の 8 項目をヒアリングし、plan doc を生成してから Phase 1 以降に進みます。

## 必要な前提プラグイン

| プラグイン | 役割 |
|---|---|
| `superpowers` | 計画・並列エージェント・検証のサブスキル提供 |
| `empirical-prompt-tuning` | Phase 4 のブラインド評価エンジン |

## インストール

```
/plugin install superpowers
/plugin install empirical-prompt-tuning
/plugin marketplace add https://github.com/oreguchi/skill-conversion.git
/plugin install skill-conversion@skill-conversion-marketplace
```

## どういう人に向いているか

- **既存スキルの多言語展開をしたい**プラグイン作者
- **別エージェント向けにポート**したい個人/チーム
- **「なぜこの記述が入っているか」を audit できる**変換プロセスを求める企業
- **品質 > 速度**を許容でき、レビューを複数回回す時間がある人

## 向いていないケース

- **ゼロからスキルを新規作成したい** → `superpowers:writing-skills` を使ってください
- **一箇所だけ直したい**（誤字修正など）→ 直接編集したほうが速いです
- **非スキル文書の翻訳** → このプラグインはスキル形式に特化しています
